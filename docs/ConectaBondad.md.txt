# ConectaBondad.org

# Product Requirements Document (PRD)

**Versión:** 1.0 (MVP)  
**Estado:** En desarrollo

---

# 0. Visión del Producto

ConectaBondad es una plataforma digital que conecta necesidades verificadas con personas y organizaciones dispuestas a ayudar, facilitando la coordinación, la transparencia y la trazabilidad de los aportes para maximizar el impacto social.

Los tres principios que guiarán todas las decisiones del proyecto:

Simplicidad: cualquier persona debe poder usar la plataforma sin capacitación.
Transparencia: todo aporte debe ser trazable y verificable.
Confianza: las organizaciones y la información publicada deben poder ser validadas.

---

# 1. Introducción

## 1.1 Objetivo

ConectaBondad.org es una plataforma web que conecta organizaciones que requieren ayuda con personas naturales, empresas e instituciones dispuestas a colaborar.

La plataforma permitirá que organizaciones registradas publiquen requerimientos asociados a una emergencia o programa activo, mientras que colaboradores registrados podrán localizar necesidades específicas y comprometer aportes para contribuir a su solución.

El objetivo del MVP es validar este modelo de colaboración mediante una plataforma sencilla, transparente y fácil de utilizar.

---

## 1.2 Alcance del MVP

El MVP incluirá las siguientes funcionalidades:

- Sitio web público.
- Registro e inicio de sesión para Organizaciones y Colaboradores.
- Administración de usuarios.
- Administración de Organizaciones.
- Administración de Colaboradores.
- Administración de Emergencias y Programas.
- Gestión de Requerimientos.
- Gestión de Aportes.
- Dashboard administrativo.
- Búsqueda y filtrado de requerimientos.

No forman parte del MVP:

- Pasarela de pagos.
- Donaciones monetarias en línea.
- Aplicación móvil.
- Multilenguaje.
- Múltiples monedas.
- Integraciones con WhatsApp, SMS o correo masivo.
- Firma electrónica.
- Geolocalización avanzada.
- Validación automática de documentos.

# 2. Arquitectura Funcional

## 2.1 Arquitectura General

La plataforma estará organizada alrededor de tres tipos de usuarios autenticados (Administrador, Organización y Colaborador), además de un sitio público para consulta de información.

Los usuarios autenticados accederán únicamente a las funcionalidades correspondientes a su rol.

Los requerimientos publicados por las organizaciones estarán asociados a una Emergencia o Programa y podrán recibir aportes de múltiples colaboradores.

El Dashboard consolidará la información de toda la plataforma para los administradores.

## 2.2 Relación entre módulos

```

```
                         Sitio Público
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
 Registro / Login       Emergencias               Requerimientos
        │
        ▼
                 Gestión de Acceso y Usuarios
        │
 ┌──────┴───────────────┐
 │                      │
 ▼                      ▼
Organizaciones      Colaboradores
 │                      │
 │                      │
 └──────────────┬───────┘
                ▼
         Gestión de Aportes
                │
                ▼
      Dashboard y Reportes
```

## 2.3 Flujo General

1. El Administrador crea una Emergencia o Programa.
2. Una Organización se registra en la plataforma.
3. La Organización publica uno o más Requerimientos.
4. Los Requerimientos se muestran públicamente.
5. Un Colaborador se registra.
6. El Colaborador selecciona un Requerimiento.
7. El Colaborador registra un Aporte.
8. La Organización confirma la recepción del Aporte.
9. El Dashboard actualiza automáticamente los indicadores.

# 3. Modelo de Datos

## 3.1 Consideraciones Generales

El modelo de datos constituye la estructura oficial de la base de datos de ConectaBondad para el MVP.

Todas las funcionalidades descritas en este PRD deberán utilizar exclusivamente las tablas y relaciones definidas en este capítulo.

### Reglas generales

- Todas las tablas principales utilizarán una llave primaria (`id`) de tipo UUID.
- Todas las entidades principales tendrán un campo `codigo`, generado automáticamente por el sistema y visible para los usuarios.
- Las relaciones entre tablas se realizarán mediante UUID.
- Ninguna contraseña será almacenada en texto plano.
- Los documentos y archivos se almacenarán mediante una URL.
- Ningún registro será eliminado físicamente; cuando sea necesario se modificará su estado (`status`).
- Todos los registros incluirán los campos `fecha_creacion` y `fecha_actualizacion`, administrados automáticamente por el sistema.

---

# 3.2 Tabla: Usuarios

## Objetivo

Almacenar las cuentas de acceso a la plataforma.

La tabla Usuarios contiene únicamente la información necesaria para autenticación y autorización.

La información propia de Organizaciones y Colaboradores se almacena en sus respectivas tablas.

| email_usuario | Texto (150) | Sí | Sí | Correo utilizado para iniciar sesión |

| tipo_usuario | Enum | Sí | No | Administrador, Organización o Colaborador |
| organizacion_id | UUID | No | No | Organización asociada |

---

## Relaciones

- Un Usuario puede estar asociado a una Organización.
- Un Usuario puede estar asociado a un Colaborador.
- Un Usuario Administrador no estará asociado a ninguna Organización ni Colaborador.

---

## Reglas de Negocio

- El correo electrónico debe ser único.
- La contraseña nunca se almacenará en texto plano.
- El sistema almacenará únicamente el hash de la contraseña.
- Solo podrá existir un Usuario asociado a una Organización durante el MVP.
- Solo podrá existir un Usuario asociado a un Colaborador durante el MVP.
- Un Usuario no podrá estar asociado simultáneamente a una Organización y a un Colaborador.
- Los usuarios no se eliminan físicamente; únicamente cambian de estado.

# 3.5 Tabla: Requerimientos

## Objetivo

Almacenar las necesidades publicadas por las Organizaciones dentro de una Campaña.

Un Requerimiento representa una necesidad específica que puede recibir aportes de uno o varios Colaboradores hasta alcanzar su cobertura total.

---

## Estructura

| Campo | Tipo | Obligatorio | Editable | Descripción |
|--------|------|-------------|----------|-------------|
| id | UUID | Sí | No | Llave primaria |
| codigo | Texto (20) | Sí | No | Código visible. Ejemplo: REQ-000001 |
| campaña_id | UUID | Sí | No | Campaña a la que pertenece |
| organizacion_id | UUID | Sí | No | Organización propietaria |
| nombre | Texto (200) | Sí | Sí | Nombre del requerimiento |
| descripcion | Texto | Sí | Sí | Descripción detallada |
| pais_id | UUID | Sí | No | Se hereda de la Campaña |
| estado_provincia_id | UUID | Sí | Sí | Estado o provincia donde existe la necesidad |
| ciudad_id | UUID | Sí | Sí | Ciudad |
| direccion | Texto | Sí | Sí | Dirección exacta |
| tipo_requerimiento_id | UUID | Sí | Sí | Tipo de requerimiento |
| cantidad_requerida | Decimal | Sí | Sí | Cantidad requerida |
| unidad_medida_id | UUID | Sí | Sí | Unidad de medida |
| cantidad_comprometida | Decimal | Sí | No | Calculada automáticamente |
| cantidad_entregada | Decimal | Sí | No | Calculada automáticamente |
| prioridad_id | UUID | Sí | Sí | Prioridad |
| fecha_requerida | Fecha | No | Sí | Fecha límite deseada |
| status | Enum | Sí | No | Borrador, Activo, Parcialmente Cubierto, Completado, Suspendido o Cancelado |
| fecha_creacion | Timestamp | Sí | No | Fecha de creación |
| fecha_actualizacion | Timestamp | Sí | No | Fecha de actualización |

---

## Relaciones

- Un Requerimiento pertenece a una Organización.
- Un Requerimiento pertenece a una Campaña.
- Un Requerimiento pertenece a un Tipo de Requerimiento.
- Un Requerimiento pertenece a una Prioridad.
- Un Requerimiento puede recibir múltiples Aportes.

---

## Reglas de Negocio

- Solo una Organización autenticada podrá crear Requerimientos.
- La Organización únicamente podrá crear Requerimientos para Campañas Activas.
- El país del Requerimiento será el mismo de la Campaña y no podrá modificarse.
- La cantidad comprometida será la suma de los Aportes comprometidos.
- La cantidad entregada será la suma de los Aportes entregados.
- Un Requerimiento cambiará automáticamente de estado:
  - Activo → Parcialmente Cubierto cuando exista al menos un aporte.
  - Parcialmente Cubierto → Completado cuando la cantidad entregada alcance o supere la cantidad requerida.
- Un Requerimiento Completado no podrá recibir nuevos Aportes.
- Los Requerimientos no se eliminan físicamente.

---

# 3.6 Tabla: Aportes

## Objetivo

Almacenar los compromisos de ayuda realizados por los Colaboradores para cubrir total o parcialmente un Requerimiento.

Un Requerimiento puede recibir múltiples Aportes y un Colaborador puede realizar múltiples Aportes.

Los Aportes constituyen la base para calcular automáticamente el avance de cada Requerimiento.

---

## Estructura

| Campo | Tipo | Obligatorio | Editable | Descripción |
|--------|------|-------------|----------|-------------|
| id | UUID | Sí | No | Llave primaria |
| codigo | Texto (20) | Sí | No | Código visible. Ejemplo: APO-000001 |
| requerimiento_id | UUID | Sí | No | Requerimiento asociado |
| colaborador_id | UUID | Sí | No | Colaborador que realiza el aporte |
| cantidad_comprometida | Decimal | Sí | Sí | Cantidad comprometida |
| cantidad_entregada | Decimal | Sí | Sí | Cantidad efectivamente entregada |
| fecha_compromiso | Fecha | Sí | No | Fecha del compromiso |
| fecha_entrega_estimada | Fecha | No | Sí | Fecha estimada de entrega |
| fecha_entrega_real | Fecha | No | No | Fecha confirmada de entrega |
| observaciones_colaborador | Texto | No | Sí | Comentarios del colaborador |
| observaciones_organizacion | Texto | No | Sí | Comentarios de la organización |
| status | Enum | Sí | No | Comprometido, En coordinación, Entregado Parcialmente, Entregado, Cancelado |
| fecha_creacion | Timestamp | Sí | No | Fecha de creación |
| fecha_actualizacion | Timestamp | Sí | No | Fecha de actualización |

---

## Relaciones

- Un Aporte pertenece a un único Requerimiento.
- Un Aporte pertenece a un único Colaborador.
- Un Requerimiento puede recibir múltiples Aportes.
- Un Colaborador puede realizar múltiples Aportes.

---

## Reglas de Negocio

- Solo un Colaborador autenticado podrá registrar Aportes.
- Solo podrán registrarse Aportes sobre Requerimientos Activos o Parcialmente Cubiertos.
- La cantidad comprometida deberá ser mayor que cero.
- La cantidad entregada nunca podrá ser mayor que la cantidad comprometida.
- El estado del Aporte cambiará automáticamente de acuerdo con su avance:
  - Comprometido
  - En coordinación
  - Entregado Parcialmente
  - Entregado
  - Cancelado
- Al registrar o actualizar un Aporte, el sistema recalculará automáticamente:
  - Cantidad comprometida del Requerimiento.
  - Cantidad entregada del Requerimiento.
  - Porcentaje de cobertura del Requerimiento.
- Los Aportes no se eliminan físicamente.

## Funcionalidades

### Iniciar sesión

El sistema deberá:

- Validar correo electrónico.
- Validar contraseña.
- Validar tipo de usuario.
- Validar estado del usuario.

Si la autenticación es correcta:

- Administrador → Dashboard Administrativo.
- Organización → Gestión de Requerimientos.
- Colaborador → Gestión de Aportes.

---

### Registrarme

Si el tipo seleccionado es Organización:

Abrir módulo:

**Gestión de Organizaciones**

Si el tipo seleccionado es Colaborador:

Abrir módulo:

**Gestión de Colaboradores**

Los Administradores no podrán registrarse desde la interfaz.

---

### Recuperar contraseña

El sistema permitirá recuperar la contraseña mediante correo electrónico.

Flujo:

1. El usuario selecciona **¿Olvidaste tu contraseña?**
2. Ingresa su correo electrónico.
3. El sistema valida que exista una cuenta activa.
4. El sistema genera un enlace temporal de recuperación.
5. Envía el enlace al correo registrado.
6. El usuario accede al enlace.
7. Define una nueva contraseña.
8. El sistema actualiza el `password_hash`.
9. Se muestra un mensaje confirmando el cambio exitoso.
10. El usuario podrá iniciar sesión con la nueva contraseña.

---

### Cerrar sesión

Finaliza la sesión y retorna al Portal Público.

## Validaciones

- El correo electrónico debe existir.
- La contraseña debe coincidir.
- El tipo de usuario seleccionado debe coincidir con el registrado.
- El usuario debe encontrarse en estado **Activo**.
- El enlace de recuperación deberá ser temporal y de un solo uso.
- La nueva contraseña deberá cumplir la política de seguridad definida por la plataforma.
- La confirmación de contraseña deberá coincidir.

Si ocurre un error de autenticación se mostrará el mensaje:

> Usuario o contraseña incorrectos.

No indicar cuál de los dos campos produjo el error.

Para recuperación de contraseña, si el correo no existe o pertenece a un usuario inactivo, mostrar un mensaje genérico como:

> Si existe una cuenta asociada a este correo, recibirás un enlace para restablecer tu contraseña.

Esto evita revelar si un correo electrónico está registrado en la plataforma.

---

## Reglas de Negocio
